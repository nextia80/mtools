# gc 명령어 스펙 개편 및 일정 수정/삭제 정리

## 개요

기존 `gc lc` / `gc lcb` 중심 구조를 단순화하고, 일정 **조회·추가·수정·삭제**를 하나의 명령 체계로 통합했다.

주요 변경:

1. **`gc` 명령 스펙 재정의** — `gc l`, `gc a`, `gc m`, `gc r`, `gc c` 통합
2. **일정 식별 키 도입** — 목록 순번 대신 `캘린더-MMDD-순번` (예: `1-0618-2`)
3. **백엔드 API 확장** — 다중 캘린더 조회, 일정 생성/수정/삭제
4. **버그 수정** — `-c` 파싱, CORS `PATCH`, 기본 캘린더 자동 선택 등
5. **GitHub 업로드** — `main` 브랜치에 커밋·push 완료

커밋: `dbd9e5e` — `gc 명령어 및 캘린더 기능 개발`

## 1. gc 명령어 스펙 (최종)

### 조회

| 명령어 | 설명 |
|--------|------|
| `gc` | 오늘 일정 (전체 캘린더) |
| `gc l` | 오늘 기준 1주일 일정 (7일) |
| `gc l <YYYYMMDD>` | 특정일 일정 |
| `gc l <YYYYMMDD>~<YYYYMMDD>` | 특정 기간 일정 |
| `gc l -c <n>[,<n>]` | 이름 앞 번호 기준 캘린더 필터 조회 |

### 캘린더 목록

| 명령어 | 설명 |
|--------|------|
| `gc c` | 내 캘린더 목록 (이름 앞 번호 = 키) |

키 규칙: Google 캘린더 이름이 `1. (개인) 흔적` 형태이면 키는 `1`.

### 일정 추가

| 명령어 | 설명 |
|--------|------|
| `gc a "<내용>"` | 오늘 종일 일정 추가 |
| `gc a "<내용>" <YYYYMMDD>` | 특정일 종일 일정 추가 |
| `gc a "<내용>" <날짜> <HHmm~HHmm>` | 시간 범위 일정 |
| `gc a "<내용>" <날짜> <HHmm+nh>` | 시작 시각 + 길이 (예: `1900+1.5h`) |
| `gc a "<내용>" ... -c <n>` | 특정 캘린더에 추가 (`-c`는 명령어 어디에나 가능) |

`-c` 미지정 시 `gc c` 목록에서 기본 캘린더를 자동 선택한다.

### 일정 수정

| 명령어 | 설명 |
|--------|------|
| `gc m <키> -t "<제목>"` | 제목 수정 |
| `gc m <키> -d "<메모>"` | 메모 수정 |
| `gc m <키> -c <n>` | 다른 캘린더로 이동 |

### 일정 삭제

| 명령어 | 설명 |
|--------|------|
| `gc r <키>` | 일정 삭제 |

키에 캘린더 번호가 포함되므로 `gc r`에는 `-c` 옵션이 없다.

### 사용 예

```text
> gc
> gc l
> gc l 20260618
> gc l 20260618~20260625
> gc l -c 1,4
> gc c
> gc a "팀 미팅" 20260618 1400~1530 -c 6
> gc m 6-0618-1 -t "주간 회의"
> gc r 6-0618-2
```

## 2. 일정 식별 키 (`캘린더-MMDD-순번`)

기존 목록 순번 `[1]`, `[2]`는 조회 범위가 바뀌면 달라져서 수정/삭제에 부적합했다.
이를 **복합 키**로 대체했다.

### 형식

```
{캘린더키}-{MMDD}-{순번}
```

예: `1-0618-2` → 1번 캘린더, 6월 18일, 그날 2번째 일정

### 순번 부여 규칙 (캘린더 + 날짜별)

1. 종일 일정 우선
2. 시작 시각 오름차순
3. 제목 오름차순

### 목록 출력 예

```text
[1-0618-1]   2026-06-18 종일 | 1. (개인) 흔적 | 회의 준비
[1-0618-2]   2026-06-18 08:38 | 1. (개인) 흔적 | test2

[6-0619-1]   2026-06-19 09:00 | 6. (회사) 업무스케줄 | 주간 회의
```

### 클릭 연동

- 일정 행의 `clickValue` = 복합 키 (`1-0618-2`)
- 터미널에서 일정 행 클릭 시 입력창에 `gc m 1-0618-2` / `gc r 1-0618-2` 형태로 채움

### 키 해석 (`resolveEventReference`)

1. 최근 `gc l` 결과 캐시(`lastListedEvents`)에서 조회
2. 없으면 키에서 캘린더·날짜를 파싱해 해당 일 API 재조회 후 순번 재계산
3. MMDD의 연도는 `getTodayYmd()` 기준 현재 연도 사용 (연말/연초 경계는 제한 사항)

## 3. 백엔드 변경 (`mtools-api`)

### API 엔드포인트

| 메서드 | 경로 | 설명 |
|--------|------|------|
| `GET` | `/api/calendar/events` | 일정 조회 (`calendarKey`, `calendarKeys` 쿼리 지원) |
| `POST` | `/api/calendar/events` | 일정 생성 |
| `PATCH` | `/api/calendar/events/{eventId}` | 일정 수정 |
| `DELETE` | `/api/calendar/events/{eventId}?calendarKey=n` | 일정 삭제 |
| `GET` | `/api/calendar/calendars` | 캘린더 목록 |

### 주요 파일

| 파일 | 변경 내용 |
|------|-----------|
| `GoogleCalendarService.java` | 다중 캘린더 조회, 생성/수정/삭제, 이름 접두 숫자로 캘린더 매칭 |
| `CalendarController.java` | `PATCH`/`DELETE` 추가, `calendarKeys` 파라미터 |
| `CreateCalendarEventRequest.java` | `calendarKey`, `endTime`, `durationMinutes` nullable |
| `UpdateCalendarEventRequest.java` | **신규** — `calendarKey`, `summary`, `description`, `newCalendarKey` |
| `CalendarEvent.java` | 응답에 `calendarKey` 필드 추가 |
| `CorsConfig.java` | `PATCH` 메서드 허용 추가 |

### CORS 수정

`gc m` 실행 시 `Failed to fetch`가 발생했던 원인은 CORS에 `PATCH`가 없었기 때문.

```java
.allowedMethods("GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS")
```

백엔드 재시작 후 반영된다.

## 5. 수정한 버그

| 증상 | 원인 | 해결 |
|------|------|------|
| `gc l -c 4,5` 무시됨 | 단일 키만 전달 | 캘린더별 개별 fetch 후 병합 |
| 캘린더 키 불일치 | 목록 인덱스 사용 | 이름 접두 숫자(`1.`)를 키로 통일 |
| `gc` vs `gc l` 범위 혼동 | 동일 처리 | `gc`=오늘, `gc l`=1주일 |
| `gc a "test"` 실패 | `calendarKey` 없음 | `-c` 미지정 시 기본 캘린더 자동 선택 |
| `gc a ... -c 1` 실패 | 따옴표 파싱 후 `-c` 누락 | 전체 문자열에서 `-c` 선추출 |
| 추가 성공 메시지 `[undefined]` | API 응답에 `calendarKey` 없음 | 캘린더 이름 접두에서 fallback |
| `gc m` → `Failed to fetch` | CORS에 `PATCH` 없음 | `CorsConfig`에 `PATCH` 추가 |

## 6. 이전 스펙과의 차이

| 이전 (6/16 기준) | 현재 |
|------------------|------|
| `gc lc <키>` | `gc l -c <키>` |
| `gc lcb <키> <시작> <끝>` | `gc l <시작>~<끝> -c <키>` |
| `gc ad` / `gc at` (별도 서브명령) | `gc a` 하나로 통합 (날짜·시간 인자로 구분) |
| 목록 순번 `[1]`, `[2]` | 복합 키 `[1-0618-2]` |
| 수정/삭제 없음 | `gc m`, `gc r` 지원 |

## 9. 알려진 제한 사항

1. **연도 경계** — 일정 키의 `MMDD`는 현재 연도 기준. 연말/연초 일정은 키만으로 연도 구분 불가.
2. **`UpdateCalendarEventRequest.calendarKey`** — primitive `int`라 null 필드 PATCH 시 400 가능성 있음 (추후 `Integer` 검토).
3. **백엔드 재시작 필요** — CORS·API 변경 반영 시 `bootRun` 재기동 필요.
