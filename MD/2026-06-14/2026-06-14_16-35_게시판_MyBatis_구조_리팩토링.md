# 게시판 MyBatis 구조 리팩토링

## 작업 배경

게시판 기능이 처음에는 `com.mtools.api` 루트 패키지 안에서 `BoardController`, `BoardService`, `BoardPost`, `BoardPostRequest`로 구성되어 있었다.

기능이 늘어나면 모든 API 클래스가 한 패키지에 모이게 되고, SQL도 서비스 코드 안에 섞이게 된다. 그래서 게시판 기능부터 먼저 `controller / service / dao / cmd` 구조로 나누고, DB 쿼리는 MyBatis Mapper XML로 분리했다.

## 변경한 구조

기존 구조는 아래처럼 루트 패키지에 게시판 관련 클래스가 바로 위치했다.

```text
com.mtools.api
├── BoardController.java
├── BoardService.java
├── BoardPost.java
└── BoardPostRequest.java
```

변경 후에는 게시판 기능을 `board` 패키지 아래로 모았다.

```text
com.mtools.api.board
├── cmd
│   └── BoardPostRequest.java
├── controller
│   └── BoardController.java
├── dao
│   └── BoardDao.java
└── service
    ├── BoardPost.java
    └── BoardService.java
```

Mapper XML은 리소스 영역에 따로 분리했다.

```text
src/main/resources/mapper/board/BoardMapper.xml
```

## MyBatis 도입

Spring Boot 4 계열에 맞춰 MyBatis starter를 추가했다.

```gradle
implementation 'org.mybatis.spring.boot:mybatis-spring-boot-starter:4.0.1'
```

그리고 `application.properties`에 Mapper XML 위치를 설정했다.

```properties
mybatis.mapper-locations=classpath:mapper/**/*.xml
```

## DAO와 Mapper 분리

`BoardDao`는 Java 코드에서 호출할 메서드만 정의한다.

```java
@Mapper
public interface BoardDao {
	List<BoardPost> findAll();
	BoardPost findById(@Param("idBoard") long idBoard);
	Long nextId();
	int insert(@Param("idBoard") long idBoard, @Param("title") String title, @Param("text") String text);
	int update(@Param("idBoard") long idBoard, @Param("title") String title, @Param("text") String text);
	int delete(@Param("idBoard") long idBoard);
}
```

실제 SQL은 `BoardMapper.xml`로 옮겼다.

```xml
<mapper namespace="com.mtools.api.board.dao.BoardDao">
	<select id="findAll" resultMap="BoardPostMap">
		SELECT
			id_board,
			st_title,
			st_text,
			dt_insert::text AS inserted_at,
			dt_update::text AS updated_at
		FROM t_board
		WHERE COALESCE(yn_use, 'Y') != 'N'
		ORDER BY
			COALESCE(dt_update, dt_insert) DESC NULLS LAST,
			id_board DESC NULLS LAST
	</select>
</mapper>
```

이제 서비스는 SQL을 직접 들고 있지 않고, `BoardDao`를 통해 게시글 조회, 등록, 수정, 삭제를 처리한다.

## 서비스 역할 정리

`BoardService`는 비즈니스 흐름만 담당하도록 정리했다.

- 게시글 목록 조회는 `boardDao.findAll()` 호출
- 등록 시 `boardDao.nextId()`로 다음 ID를 구한 뒤 insert
- 수정 시 변경된 행이 없으면 예외 처리
- 삭제는 기존처럼 `yn_use = 'N'`으로 소프트 삭제
- 등록/수정 후에는 `findById`로 최신 데이터를 다시 조회

## 유지한 API

프론트엔드와 연결된 API 경로는 그대로 유지했다.

```text
GET    /api/boards
POST   /api/boards
PUT    /api/boards/{idBoard}
DELETE /api/boards/{idBoard}
```

그래서 프론트엔드 코드는 수정하지 않아도 기존 게시판 화면이 그대로 동작한다.

## 검증 결과

백엔드 컴파일을 확인했다.

```shell
./gradlew compileJava
```

결과는 성공이었다.

```text
BUILD SUCCESSFUL
```

이후 백엔드를 새로 실행하고 게시판 목록 API도 확인했다.

```shell
curl -s http://localhost:8080/api/boards
```

응답으로 기존 게시글 목록이 정상 조회되었다.

## 정리

이번 작업으로 게시판 기능은 단순히 동작하는 수준에서 한 단계 정리된 구조가 되었다.

컨트롤러는 HTTP 요청과 응답을 담당하고, 서비스는 게시판의 처리 흐름을 담당한다. DAO는 DB 접근 메서드만 정의하고, 실제 SQL은 Mapper XML에서 관리한다.

다음 단계에서는 같은 패턴을 `member` 기능에도 적용할 수 있다. Markdown 파일 관리 기능은 DB 중심 기능이 아니기 때문에, 같은 구조를 적용하더라도 DAO보다는 파일 저장소 역할을 분리하는 방식이 더 자연스럽다.
