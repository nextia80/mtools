# Vue 메뉴별 컴포넌트 분리

## 작업 배경

`App.vue` 한 파일에 화면, 상태, API 호출, 스타일이 모두 들어 있어서 파일이 약 2,000줄 이상으로 비대해졌다.

메뉴가 늘어날수록 수정 범위가 커지고, 특정 화면만 고치기도 어려운 구조였다. 그래서 메뉴별로 Vue 컴포넌트를 분리하고, 공통 타입과 스타일도 별도 파일로 빼서 관리하기 쉽게 정리했다.

## 변경 전 구조

```text
mtools-ui/src/
└── App.vue
    ├── script (상태 + API + 로직 전부)
    ├── template (사이드바 + 모든 메뉴 화면)
    └── style (1,200줄 이상 CSS)
```

`App.vue` 안에서 아래가 한꺼번에 처리되고 있었다.

- 좌측 메뉴와 일자별 MD 중첩 메뉴
- 홈 / 게시판 / 회원관리 / API 스웨거 / API 테스트 / MD 뷰어
- 게시판 CRUD, MD 편집, API 호출 등 비즈니스 로직
- 전체 UI 스타일

## 변경 후 구조

```text
mtools-ui/src/
├── App.vue                 (상태 관리 + 화면 조립)
├── types.ts                (공통 타입)
├── assets/
│   ├── main.css
│   └── app.css             (기존 App.vue 스타일)
└── components/
    ├── SidebarMenu.vue     (좌측 메뉴)
    ├── BoardView.vue       (게시판)
    ├── MemberView.vue      (회원관리)
    ├── SwaggerView.vue     (API 스웨거)
    ├── ApiTestView.vue     (API 테스트)
    └── DocsView.vue        (일자별 MD 뷰어)
```

## 각 파일 역할

### App.vue

`App.vue`는 이제 **앱 전체 상태와 화면 전환**을 담당한다.

- `activeView`로 현재 메뉴 관리
- API 호출, 데이터 로딩, 저장 로직 유지
- 분리된 컴포넌트에 props로 데이터 전달
- 컴포넌트에서 올라온 이벤트를 받아 처리

```text
App.vue
  ├── SidebarMenu
  └── main
        ├── 홈 (빈 화면)
        ├── BoardView
        ├── MemberView
        ├── SwaggerView
        ├── ApiTestView
        └── DocsView
```

### types.ts

여러 컴포넌트에서 쓰는 타입을 한곳으로 모았다.

```typescript
export type MdFile = { ... }
export type BoardPost = { ... }
export type ActiveView = 'home' | 'board' | 'member' | 'swagger' | 'api' | 'docs'
export type MdViewMode = 'review' | 'edit'
```

### SidebarMenu.vue

좌측 메뉴 전체를 담당한다.

- 홈 / 게시판 / 회원관리 / API 스웨거 / API 테스트
- 일자별 LOG 메뉴와 날짜별 MD 파일 목록
- 새로고침 버튼

`App.vue`와는 이벤트로 연결한다.

```text
props: activeView, mdFiles, mdFilesByDate, ...
events: setActiveView, refreshMdFiles, openDateGroup, selectMdFile
```

### BoardView.vue

게시판 화면만 담당한다.

- 게시글 작성/수정 폼
- 게시글 목록 표시
- 수정/삭제 버튼

데이터와 저장 로직은 `App.vue`에 두고, 화면 표현만 분리했다.

### ApiTestView.vue

API 호출 테스트 화면.

- End point 입력
- JSON 입력
- 실행 버튼과 결과 표시

### SwaggerView.vue

Swagger UI iframe 화면.

### DocsView.vue

일자별 MD 뷰어 화면.

- 리뷰 / 편집 모드 전환
- Markdown 렌더링
- 편집기 포커스 처리

### app.css

기존 `App.vue`에 있던 대량의 CSS를 `src/assets/app.css`로 분리했다.

`main.css`에서 import 한다.

```css
@import './base.css';
@import './app.css';
```

## props / events 흐름 예시

게시판을 기준으로 보면 흐름은 아래와 같다.

```text
App.vue
  posts, title, text, isLoading ...
    ↓ props
BoardView.vue
  등록/수정/삭제 버튼 클릭
    ↓ events
App.vue
  saveBoardPost(), deleteBoardPost() 실행
```

즉 **로직은 App.vue**, **화면은 각 View 컴포넌트**가 담당하는 구조다.

`App.vue` 자체는 약 73% 줄어들었다. 스타일을 `app.css`로 빼면서 전체 코드는 파일 단위로 나뉘었지만, 한 파일에 몰려 있던 부담은 크게 줄었다.

## 검증 결과

프론트엔드 빌드와 타입체크를 수행했다.

```shell
npm --prefix mtools-ui run type-check
npm --prefix mtools-ui run build-only
```

둘 다 성공했다.

## 정리

이번 작업으로 Vue 프론트엔드는 "한 파일에 모든 것" 구조에서 "메뉴별 컴포넌트 + 공통 타입 + 공통 CSS" 구조로 바뀌었다.

앞으로 메뉴나 화면을 수정할 때 `App.vue` 전체를 보지 않고, 해당 View 컴포넌트만 보면 된다.

다음 단계로는 API 로직을 `composables`로 더 분리하면 `App.vue`를 더 가볍게 만들 수 있다.

```text
예시
composables/useBoard.ts
composables/useMdFiles.ts
composables/useApiTest.ts
```

이렇게 하면 `App.vue`는 화면 조립과 상태 연결만 담당하는 형태로 더 정리할 수 있다.
