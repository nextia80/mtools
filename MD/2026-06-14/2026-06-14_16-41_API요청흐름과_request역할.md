# API 요청 흐름과 request 역할 정리

## request는 무엇인가

`request`는 프론트엔드에서 백엔드 API로 보낸 입력값을 담는 객체다.

예를 들어 게시글을 저장할 때 프론트엔드에서는 아래처럼 JSON을 보낸다.

```json
{
  "title": "제목",
  "text": "내용"
}
```

백엔드에서는 이 JSON을 `BoardPostRequest` record로 받는다.

```java
@PostMapping("/api/boards")
public BoardPost createBoard(@RequestBody BoardPostRequest request) {
	return boardService.create(request);
}
```

여기서 `@RequestBody`는 HTTP 요청 body에 들어온 JSON을 Java 객체로 변환해준다.

즉 `BoardPostRequest`는 DB 조회 결과를 담는 객체가 아니라, 사용자가 화면에서 입력해서 서버로 보낸 값을 담는 객체다.

## 게시글 등록 기준 전체 흐름

게시글 등록 API를 기준으로 보면 흐름은 아래와 같다.

```text
Vue 화면
  ↓
POST /api/boards
  ↓
BoardController
  ↓
BoardPostRequest
  ↓
BoardService
  ↓
BoardDao
  ↓
BoardMapper.xml
  ↓
PostgreSQL t_board
```

## 각 폴더의 역할

### controller

`controller`는 API의 입구다.

프론트엔드에서 `/api/boards`로 요청을 보내면 `BoardController`가 먼저 요청을 받는다.

```java
@PostMapping("/api/boards")
public BoardPost createBoard(@RequestBody BoardPostRequest request) {
	return boardService.create(request);
}
```

컨트롤러는 요청을 받고, 필요한 값을 request 객체로 받은 다음, 실제 처리는 service에 넘긴다.

### request

`request`는 프론트엔드에서 들어온 요청값을 담는다.

게시글 등록/수정의 경우 `title`, `text`가 여기에 들어간다.

```java
public record BoardPostRequest(
		String title,
		String text
) {
}
```

이 객체는 "사용자가 서버에 보낸 값"을 표현한다.

### service

`service`는 처리 흐름을 담당한다.

예를 들어 게시글 등록 시에는 아래 흐름을 처리한다.

- 다음 게시글 ID를 구한다.
- 게시글을 저장한다.
- 저장된 게시글을 다시 조회한다.
- 조회한 결과를 컨트롤러에 돌려준다.

서비스는 SQL을 직접 들고 있지 않고, DB 처리는 DAO에 위임한다.

### dao

`dao`는 DB 접근 창구다.

서비스가 `boardDao.insert(...)`, `boardDao.findAll()` 같은 메서드를 호출하면 MyBatis가 이 메서드와 Mapper XML의 SQL을 연결한다.

```java
@Mapper
public interface BoardDao {
	List<BoardPost> findAll();
	Long nextId();
	int insert(long idBoard, String title, String text);
}
```

### mapper xml

`mapper xml`은 실제 SQL을 관리하는 파일이다.

```xml
<insert id="insert">
	INSERT INTO t_board (
		id_board,
		st_title,
		st_text,
		yn_use,
		id_insert,
		dt_insert
	)
	VALUES (
		#{idBoard},
		#{title},
		#{text},
		'Y',
		1,
		CURRENT_TIMESTAMP
	)
</insert>
```

Java 코드에서는 SQL을 직접 작성하지 않고, Mapper XML에서 SQL을 관리한다.

## 응답 객체와 request의 차이

`request`는 프론트엔드에서 백엔드로 들어오는 값이다.

반대로 `BoardPost` 같은 객체는 백엔드에서 프론트엔드로 돌려주는 값이다.

```text
request
프론트엔드 → 백엔드
사용자가 입력한 값

response 또는 dto
백엔드 → 프론트엔드
DB에서 조회하거나 처리된 결과
```

현재 프로젝트에서는 `BoardPost`가 `service` 패키지에 있지만, 나중에 더 깔끔하게 정리하려면 `response` 또는 `dto` 패키지로 분리할 수 있다.

## 한 줄 요약

```text
request = 프론트에서 들어오는 입력값
controller = API 입구
service = 처리 흐름
dao = DB 접근 메서드
mapper = 실제 SQL
DB = 최종 저장소
```

이번 구조를 기준으로 기능을 늘리면, 각 클래스가 어떤 역할을 하는지 명확해지고 SQL과 Java 로직이 섞이지 않아 유지보수가 쉬워진다.
