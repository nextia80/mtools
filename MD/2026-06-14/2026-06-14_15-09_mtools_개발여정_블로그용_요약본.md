# Vue 3와 Spring Boot로 나만의 작업 로그 도구 만들기

## 시작하며

처음에는 단순했다.

Vue 3 프로젝트를 하나 만들고, Spring Boot API와 연결해보자는 정도였다.

그런데 작업을 하다 보니 자연스럽게 욕심이 생겼다.

```text
작업 내용을 Markdown으로 남기고
그 Markdown을 다시 웹 화면에서 보고
필요하면 바로 편집하고 저장할 수 있으면 좋겠다.
```

그렇게 `mtools`라는 작은 작업 도구가 만들어지기 시작했다.

결과적으로 지금의 구조는 아래와 같다.

```text
mtools
├─ mtools-ui   : Vue 3 화면
├─ mtools-api  : Spring Boot API 서버
├─ MD          : 작업 기록 Markdown 파일
└─ PostgreSQL  : 테스트 데이터 저장소
```

이 글은 2026년 6월 13일부터 6월 14일까지, 이 도구를 처음부터 조금씩 만들어간 과정을 정리한 기록이다.

## 1. Vue 3 프로젝트부터 시작했다

첫 단계는 프론트엔드였다.

`mtools-ui`라는 이름으로 Vue 3 프로젝트를 만들었다.

```bash
npm create vue@latest mtools-ui
```

처음부터 거창한 구조를 잡기보다는, Vue 화면 하나에서 API를 호출하고 결과를 보여줄 수 있는 정도를 목표로 했다.

초반에는 Router나 Pinia 같은 기능도 고민했지만, 일단은 최대한 단순하게 시작했다.

중요한 것은 프로젝트가 커지는 것이 아니라, 화면과 API가 실제로 연결되는 흐름을 빠르게 확인하는 것이었다.

## 2. Spring Boot API 서버를 만들었다

다음은 백엔드였다.

`mtools-api`라는 Spring Boot 프로젝트를 만들고, Vue 화면에서 호출할 수 있는 API 서버 역할을 맡겼다.

예상한 전체 구조는 아래와 같았다.

```text
mtools/
  mtools-ui/   # Vue 3 프론트엔드
  mtools-api/  # Spring Boot 백엔드 API 서버
  MD/          # 작업 기록
```

Spring Boot 프로젝트는 Gradle 기반으로 생성했고, 기본 API 서버는 `8080` 포트에서 실행되도록 했다.

이 시점의 목표는 명확했다.

```text
브라우저
-> Vue 화면
-> Spring Boot API
-> JSON 응답
-> 다시 Vue 화면에 표시
```

## 3. Echo API로 프론트와 백엔드를 연결했다

첫 번째 API는 아주 단순한 Echo API였다.

Vue 화면에서 `id` 값을 보내면, Spring Boot가 받은 값을 그대로 돌려주는 기능이다.

```text
GET /api/echo?id=a
```

응답은 아래처럼 온다.

```json
{
  "id": "a"
}
```

이 API는 작지만 중요한 의미가 있었다.

프론트와 백엔드가 실제로 통신한다는 것을 확인하는 첫 연결점이었기 때문이다.

브라우저 개발 서버는 `5173` 또는 `5174` 포트에서 실행되고, Spring Boot는 `8080` 포트에서 실행된다.

그래서 CORS 설정도 함께 추가했다.

```java
@CrossOrigin(origins = {"http://localhost:5173", "http://localhost:5174"})
```

이후 Vue 화면에는 id 입력창과 API 호출 버튼을 만들고, 응답값을 화면에 표시했다.

작은 테스트였지만 이 단계에서 전체 개발 흐름이 잡혔다.

## 4. PostgreSQL을 연결했다

다음으로 PostgreSQL을 붙였다.

프로젝트 전용 계정과 데이터베이스를 아래처럼 맞췄다.

```text
database : mtools
schema   : mtools
user     : mtools
password : 123456
```

Spring Boot 설정도 PostgreSQL에 맞췄다.

```properties
spring.datasource.url=jdbc:postgresql://localhost:5432/mtools?currentSchema=mtools
spring.datasource.username=mtools
spring.datasource.password=123456
```

처음에는 복잡한 테이블이 필요하지 않았다.

`tbl_test`라는 테스트 테이블을 만들고, 모든 컬럼에 `1`이 들어간 간단한 데이터를 넣었다.

그리고 아래 API를 만들었다.

```text
GET /api/tbl-test
```

전체 흐름은 아래와 같았다.

```text
Vue 또는 브라우저
-> Spring Boot Controller
-> Service
-> PostgreSQL tbl_test 조회
-> JSON 응답
```

이 단계에서 `mtools`는 단순 화면이 아니라 데이터베이스까지 연결된 작은 웹 애플리케이션이 되었다.

## 5. 작업 기록을 Markdown으로 남기기 시작했다

프로젝트를 만들면서 자연스럽게 작업 내용을 문서로 남기기 시작했다.

처음에는 단순한 기록이었다.

```text
무엇을 만들었는지
어떤 명령어를 실행했는지
왜 그렇게 설정했는지
어떤 문제가 있었는지
어떻게 해결했는지
```

이 기록들은 `MD` 폴더 아래 날짜별로 저장했다.

```text
MD/
├─ 2026-06-13/
└─ 2026-06-14/
```

파일명도 시간과 주제를 포함하도록 했다.

```text
2026-06-14_14-57_Markdown_편집_단축키와_포커스.md
```

여기서 다음 아이디어가 나왔다.

```text
어차피 작업 기록이 Markdown으로 쌓이고 있다면,
이걸 웹 화면에서 바로 볼 수 있지 않을까?
```

## 6. MD 뷰어를 만들었다

브라우저는 보안상 로컬 프로젝트 폴더를 직접 읽을 수 없다.

그래서 Vue가 직접 `MD` 폴더를 읽는 대신, Spring Boot가 중간에서 파일을 읽어주는 구조로 만들었다.

전체 흐름은 아래와 같다.

```text
Vue 화면 접속
-> Spring Boot에 MD 파일 목록 요청
-> Spring Boot가 프로젝트 MD 폴더를 읽음
-> .md 파일 목록을 JSON으로 반환
-> Vue가 목록 표시
-> 파일 클릭
-> Spring Boot에 파일 내용 요청
-> Markdown 원문 반환
-> Vue에서 HTML로 렌더링
```

목록 API는 아래와 같다.

```text
GET /api/md-files
```

내용 API는 아래와 같다.

```text
GET /api/md-files/content?path=...
```

Spring Boot에서는 `MD` 폴더 아래의 `.md` 파일만 읽을 수 있도록 경로를 제한했다.

이 제한은 중요했다.

로컬 파일을 다루는 기능이기 때문에, 사용자가 임의 경로로 다른 파일을 읽지 못하게 해야 했다.

## 7. 화면을 좌측 메뉴 중심으로 바꿨다

처음 MD 뷰어는 단순한 카드 레이아웃이었다.

API 테스트 영역과 MD 뷰어가 한 화면에 같이 있었다.

하지만 작업 기록이 늘어나면서 목록 탐색이 중요해졌다.

그래서 화면을 좌측 메뉴 중심으로 바꿨다.

처음에는 단순히 MD 파일 목록을 왼쪽에 두었다.

이후 메뉴 구조를 아래처럼 정리했다.

```text
홈
API 테스트
일자별LOG
```

`홈`은 빈 화면으로 두었다.

`API 테스트`는 Echo API 호출 테스트 화면이다.

`일자별LOG`는 날짜별 Markdown 작업 기록 목록이다.

날짜별 목록은 아래처럼 중첩 메뉴로 구성했다.

```text
일자별LOG (16)
├─ 20260614
│  ├─ (14:57) Markdown_편집_단축키와_포커스.md
│  ├─ (14:48) Markdown_리뷰_편집_저장기능.md
│  └─ ...
└─ 20260613
   ├─ (23:31) MD뷰어_동작방식.md
   └─ ...
```

파일명은 원래 아래처럼 길었다.

```text
2026-06-14_14-30_좌측메뉴_홈_API_일자별MD_분리.md
```

하지만 날짜는 이미 폴더명에 있으므로 메뉴에서는 날짜를 제거하고 시간만 보기 좋게 바꿨다.

```text
(14:30) 좌측메뉴_홈_API_일자별MD_분리.md
```

작은 변화지만 실제 사용성은 훨씬 좋아졌다.

## 8. 날짜 폴더는 하나만 열리도록 했다

MD 파일이 많아지면 모든 날짜가 열려 있는 것이 오히려 불편했다.

그래서 가장 최근 날짜만 기본으로 열고, 다른 날짜를 클릭하면 그 날짜만 열리도록 바꿨다.

```text
처음 화면
-> 가장 최근 날짜만 열림

다른 날짜 클릭
-> 클릭한 날짜만 열림
-> 기존 날짜는 닫힘
```

또한 MD 목록을 너무 자주 불러올 필요가 없어서 자동 갱신 주기를 5초에서 1분으로 바꿨다.

새 파일을 즉시 확인하고 싶을 때는 `새로고침` 버튼을 누르면 된다.

## 9. Markdown을 편집할 수 있게 만들었다

처음에는 Markdown을 보기만 했다.

하지만 실제로 쓰다 보니 자연스럽게 다음 요구가 생겼다.

```text
읽다가 바로 고치고 싶다.
수정한 내용을 다시 파일에 저장하고 싶다.
```

그래서 문서 뷰어에 `리뷰 / 편집 / 저장` 기능을 추가했다.

```text
리뷰
- Markdown을 HTML로 렌더링해서 보여줌

편집
- Markdown 원문을 textarea에서 수정

저장
- 수정한 내용을 백엔드 API로 전송
- 실제 MD 파일에 반영
- 저장 후 리뷰 모드로 전환
```

저장 API는 아래처럼 만들었다.

```text
PUT /api/md-files/content?path=...
```

요청 body는 간단하다.

```json
{
  "content": "수정된 Markdown 내용"
}
```

백엔드에서는 기존 파일 경로 검증 로직을 그대로 사용했다.

따라서 `MD` 폴더 밖의 파일이나 `.md`가 아닌 파일은 저장할 수 없다.

## 10. 키보드 단축키를 넣었다

편집 기능이 들어가자 버튼 클릭도 조금 번거롭게 느껴졌다.

그래서 단축키를 추가했다.

```text
⌘E : 편집 모드
⌘S : 저장
⌘V : 리뷰 모드
```

특히 `⌘E`를 눌렀을 때 편집 textarea에 바로 포커스가 가도록 했다.

이제 흐름이 자연스럽다.

```text
문서 선택
-> ⌘E
-> 바로 편집
-> ⌘S
-> 저장
-> 리뷰 화면으로 전환
```

Vue에서는 `nextTick`과 textarea `ref`를 이용해서 편집기가 렌더링된 뒤 포커스를 보냈다.

```ts
const focusMdEditor = async () => {
  await nextTick()
  mdEditorRef.value?.focus()
}
```

작은 기능이지만, 실제로 계속 글을 고치면서 쓰는 도구에서는 이런 부분이 체감된다.

## 11. UI를 계속 다듬었다

기능을 만든 뒤에는 화면을 계속 다듬었다.

오른쪽 문서 영역은 화면을 꽉 채우도록 바꿨다.

편집 textarea는 처음에 너무 낮게 보였기 때문에, 뷰포트 높이를 기준으로 더 크게 잡았다.

```css
.md-viewer {
  min-height: calc(100vh - 11rem);
}

.md-editor {
  min-height: calc(100vh - 17rem);
}
```

좌측 메뉴도 계속 정리했다.

```text
메뉴 간격 줄이기
파일명이 길면 ... 처리
가로 스크롤 제거
날짜/파일 목록 세로 간격 줄이기
일자별LOG 제목과 개수, 새로고침 버튼을 한 줄로 배치
```

처음에는 기능 중심이었지만, 점점 실제로 쓰기 편한 도구에 가까워졌다.

## 12. 지금의 mtools는 어떤 도구인가?

현재 `mtools`는 단순한 예제 프로젝트가 아니다.

내가 작업하면서 남긴 Markdown 기록을 웹 화면에서 확인하고, 날짜별로 탐색하고, 바로 수정해서 다시 저장할 수 있는 작은 작업 로그 도구다.

현재 가능한 일은 아래와 같다.

```text
Vue 3 화면에서 Spring Boot API 호출
PostgreSQL 테스트 데이터 조회
MD 폴더의 Markdown 파일 목록 조회
날짜별 작업 로그 탐색
Markdown 리뷰 모드
Markdown 편집 모드
Markdown 저장
키보드 단축키로 편집/저장/리뷰 전환
```

기술적으로는 단순한 조합이다.

```text
Vue 3
Spring Boot
PostgreSQL
Markdown
파일 시스템 API
```

하지만 이 조합으로 꽤 실용적인 개인 도구가 만들어졌다.

## 마무리

이번 작업의 재미있는 점은 처음부터 완성된 설계를 가지고 시작하지 않았다는 것이다.

처음에는 Vue 프로젝트를 만들었다.

그 다음 Spring Boot API를 붙였다.

그 다음 PostgreSQL을 연결했다.

작업 기록을 Markdown으로 남기다 보니, 그 기록을 웹에서 보고 싶어졌다.

웹에서 보다 보니 좌측 메뉴가 필요해졌다.

읽다 보니 바로 편집하고 싶어졌다.

편집하다 보니 단축키와 포커스가 필요해졌다.

결국 기능은 사용하면서 자연스럽게 생겼다.

이 프로젝트는 거창한 서비스는 아니지만, 내가 일하는 방식을 조금 더 편하게 만드는 도구가 되었다.

앞으로는 여기에 검색, 태그, 새 문서 작성, 이미지 첨부 같은 기능을 붙일 수도 있을 것 같다.

하지만 지금 단계만으로도 충분히 의미가 있다.

내가 만든 도구로, 내가 만든 기록을 다시 읽고 고칠 수 있게 되었기 때문이다.
